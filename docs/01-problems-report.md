## 一、严重问题（Critical）

### 1.1 隐私声明与实际行为严重不符
**文件：** `PrivacyInfo.xcprivacy`

**问题描述：**
- 隐私文件声明采集「联系人数据」和「使用数据」用于分析（Analytics）
- 但 `app-store-metadata.md` 明确写道："We don't collect anonymous data either"
- `SharingManager.swift` 中 `generateShareCard()` 方法会读取用户通讯录（`CNContactStore`）用于分享
- `CloudSyncManager.swift` 会采集分析数据并上传

**风险：** App Store 审核极大概率因隐私声明不一致被拒绝上架。

---

### 1.2 NSUserNotificationUsageDescription 用于 iOS（平台错误）
**文件：** `Info.plist`

```xml
<key>NSUserNotificationUsageDescription</key>
<string>...</string>
```

**问题描述：**
- `NSUserNotificationUsageDescription` 是 **macOS** 的通知权限 key
- iOS 上应使用 `UNUserNotificationCenter` 框架，对应 key 是 `NSUserNotificationCenterUsageDescription`（iOS 10+ 已废弃）
- iOS 正确写法为：**直接不声明**（默认请求权限）或使用 `UNUserNotificationCenter` API
- 当前写法**完全无效**，用户在 iOS 17 上不会收到任何通知权限提示

**影响：** 推送通知功能在 iOS 上完全失效。

---

### 1.3 缺少关键 Info.plist 条目
**文件：** `Info.plist`

**缺失内容：**
- 缺少 `UIBackgroundModes`（后台模式），如果 App 需要后台刷新或推送唤醒
- 缺少 `ITSAppUsesNonExemptEncryption`（出口合规性声明），App Store Connect 要求所有 App 必须填写此字段（即使是 `false`）
- 缺少 HealthKit 相关描述 `NSHealthShareUsageDescription` / `NSHealthUpdateUsageDescription`

## 二、主要问题（Major）

## 四、架构问题（Architecture）

### 4.1 Store 注入方式不符合 Apple 最佳实践
**文件：** `App/BloomApp.swift`

```swift
ContentView()
    .environmentObject(UserStore())
    .environmentObject(PlantEngine())
```

**问题描述：**
- 使用 `@StateObject` 初始化 `PlantEngine`，但 `PlantEngine` 依赖 `UserStore`（通过 `@EnvironmentObject var userStore`）
- 初始化顺序不确定，可能导致 `userStore` 为 nil 时被访问

---

### 4.2 分享功能硬依赖 CNContactStore
**文件：** `Core/Managers/SharingManager.swift`

```swift
let contacts = try store.unifiedContacts(...)
```

**问题描述：**
- 在 iOS 17 上，读取通讯录需要 `NSContactsUserDefaults` 或 entitlement
- 若用户拒绝通讯录权限，分享功能完全失效

---

### 4.3 Widget 与主 App 数据共享机制脆弱
**文件：** `Core/Managers/WidgetDataManager.swift`

**问题描述：**
- 使用 `UserDefaults(suiteName: "group.com.waterminder.bloom")` 共享数据
- 若 App Group 配置错误，Widget 显示旧数据或崩溃

---

### 4.4 植物浇水逻辑耦合在 View 层
**文件：** `Features/Garden/GardenView.swift`

```swift
private func waterPlant(_ cup: CupType) {
    let amount = cup.defaultAmount
    waterStore.add(amount: amount, cupType: cup)
    plantEngine.water(amount: amount)
    if waterStore.isGoalMetToday {
        plantEngine.processGoalMet()
    }
    if healthManager.isAuthorized {
        Task { try? await healthManager.saveWater(amount) }
    }
    splashTrigger += 1
    UIImpactFeedbackGenerator(style: .medium).impactOccurred()
}
```

**问题描述：**
- 浇水业务逻辑分散在 View 中，应移至 PlantEngine 或 WaterStore
- 不便于单元测试

## 五、安全问题（Security）

### 5.1 内购凭证验证在客户端完成
**文件：** `Core/Managers/StoreManager.swift`

**问题描述：**
- `verifyPurchase()` 验证逻辑在设备端执行，攻击者可 hook 该方法
- 应在服务端验证 `SKPaymentTransaction.transactionReceipt`

---

### 5.2 UserDefaults 存储敏感状态
**文件：** `Core/Stores/UserStore.swift`

```swift
@AppStorage("isPro") var isPro: Bool = false
```

**问题描述：**
- `isPro` 状态存储在未加密的 UserDefaults，可被轻易修改
- 建议迁移至 Keychain 并添加服务器端验证

---

### 5.3 分享图片包含敏感信息
**文件：** `Core/Managers/SharingManager.swift`

**问题描述：**
- `generateShareCard()` 生成的图片包含用户喝水量、植物健康度等隐私数据
- 直接分享到社交平台可能泄露用户健康信息

## 六、性能问题（Performance）

### 6.1 植物动画每帧重绘
**文件：** `UI/PlantCanvas/AnimatedPlantView.swift`

### 6.2 大量通知存储在 UserDefaults
**文件：** `Core/Managers/NotificationManager.swift`

**问题描述：**
- `pendingNotifications` 使用 `UserDefaults` 存储，iOS 限制单条 Notification payload 大小

### 6.3 WaterStore records 数组无限增长
